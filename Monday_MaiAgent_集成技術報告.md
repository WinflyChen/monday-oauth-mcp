## 📋 執行摘要

該專案實現了 **Monday.com** 與 **MaiAgent** 的完整集成，通過 **MCP（Model Context Protocol）** 標準協議連接。使用者可以直接在 MaiAgent 的對話中操作 Monday.com 的 Board、Item 和文件。

**關鍵成就：**
- ✅ OAuth 2.0 完整認證流程
- ✅ 文件二進制上傳（縮圖正常顯示，已驗證）
- ✅ MCP JSON-RPC 標準集成
- ✅ 多使用者架構支持
- ✅ 生產級別部署就緒

## 🔧 第一部分：平台整合需要的條件與技術

### 1.1 Monday.com 側的要求

#### **必須條件：**

| 條件 | 說明 | 實現位置 |
|-----|------|--------|
| **OAuth App** | 需在 Monday.com 開發者後台建立應用；`CLIENT_ID` / `SECRET` 屬於 **App 層級**，只需開發者設定一次，所有使用者共用同一組，各自透過 OAuth 登入後拿到各自的 Access Token | `MONDAY_CLIENT_ID`, `MONDAY_CLIENT_SECRET` |
| **App 權限 (Scopes)** | 授予應用基本權限 | OAuth 授權頁面勾選 |
| **Redirect URL** | 必須在 App OAuth 設定中白名單 `http://localhost:3002/oauth/callback`；部署雲端後需改為 HTTPS 網址並同步更新後台設定 | Monday 開發者後台 → OAuth → Redirect URLs |
| **GraphQL v2 API** | Monday 必須開放 GraphQL API 端點 | 自動可用 |
| **File 列類型** | Monday Board 須有「文件」類型的列 | Board 設定中建立 |

#### **技術要求：**

```
Monday.com 帳戶
    ├─ 企業級或付費帳戶（提供 API 存取權限）
    ├─ 開發者權限（建立 OAuth App）
    ├─ API 配額（每分鐘請求上限）
    └─ GraphQL 端點可存取性
```

#### **當前已配置：** ✅

```env
MONDAY_CLIENT_ID=d09eae3449d5efeb0487971ff9dfe354
MONDAY_CLIENT_SECRET=1c5f6145adf23b736eb0be4e9a0db1f7
MONDAY_REDIRECT_URI=http://localhost:3002/oauth/callback
```

> **注意：** `CLIENT_ID` / `SECRET` 為 App 憑證，寫在伺服器 `.env` 即可，**不需要每個使用者各自提供**。使用者首次登入時走一次 OAuth 流程，伺服器會為其儲存個人的 Access Token（`tokens.json`），後續操作各自使用自己的 Token。

### 1.4 核心技術堆棧總結

#### **通訊協議層：**

| 協議 | 用途 | 實例 |
|-----|------|------|
| **OAuth 2.0** | Monday.com 使用者認證 | `GET /oauth/authorize` → 使用者登入 → `GET /oauth/callback?code=xxx` |
| **GraphQL v2** | Monday API 資料查詢 | `POST https://api.monday.com/v2` with GraphQL query |
| **multipart/form-data** | 文件二進制上傳 | `POST /api/upload` with `form-data` body |
| **JSON-RPC 2.0** | MaiAgent 工具呼叫 | `POST /` with `{"jsonrpc": "2.0", "method": "tools/call"}` |

## ⚠️ 第二部分：平台整合的限制與挑戰

### 2.1 Monday.com API 的設計限制

#### **限制 1：File 欄位上傳困難**

**問題：** Monday 的「文件」類型列不能直接透過 GraphQL 接收二進制資料。

**原因：** Monday API 的設計限制
- ❌ `add_file_to_item(file_url: "...")` 只接受 **URL 字串**，不是二進制
- ❌ 無法直接上傳到 File 列，必須先上傳到某處，再儲存 URL

**專案中的解決方案：** ✅

```javascript
// 呼叫 Monday 的專用端點
POST https://api.monday.com/v2/file

mutation add_file($file: File!) {
    add_file_to_column(
        item_id: 12345,
        column_id: "file_mkzrr5mp",
        file: $file              // 二進制資料
    )
}
```

---

#### **限制 2：Board 建立權限問題**

**問題：** 不是所有使用者都能透過 API 建立 Board。

**原因：** Monday 工作區的權限設定

**限制：**
- 需要工作區權限 `boards:write`
- 某些工作區可能禁用了 API 建立功能
- 企業版才支援完整的 API 權限

**當前方案：** 使用者需在 Monday UI 中手動建立 Board，然後透過 API 來管理 Item。

---

#### **限制 3：API 速率限制**

**問題：** Monday 對 API 請求有限流。

**限制詳情：**
- Monday 免費版：100 請求/分鐘
- Monday 付費版：1000+ 請求/分鐘
- 超過限制會傳回 429 Too Many Requests

**專案中的風險：** ⚠️

當前程式碼沒有實現請求佇列或速率限制器，如果大量使用者同時上傳，可能觸發限流。

**建議改進：** 需添加請求佇列機制

---

### 2.2 MaiAgent 集成的限制

#### **限制 1：工具之間不能呼叫**

**問題：** MCP 工具是單向的，一個工具輸出不能直接是另一個工具的輸入。

**限制：**
```
使用者 → monday_create_board → Board ID
但這個 Board ID 不能自動傳給 monday_create_item

需要使用者再次輸入：「在剛才建立的 Board 中新增項目」
```

**影響：** 需要多輪對話

---

#### **限制 2：文件路徑處理困難**

**問題：** MaiAgent 運行在雲端，本機檔案無法直接存取。

**限制：**
- MaiAgent 無法讀取使用者本機硬碟的檔案
- 必須由使用者先上傳到網路存儲
- 或在對話中描述檔案位置

**可行解決方案：**
1. 先把檔案上傳到 MaiAgent 存儲
2. MCP 工具讀取已上傳的檔案
3. 然後上傳到 Monday

---

#### **限制 3：實時性與非同步處理**

**問題：** MaiAgent 對話需要等待 MCP 工具的同步回應。

**限制：**
```
MaiAgent 呼叫工具 → 等待 MCP 回覆 → 回覆超時 (通常 30-60 秒)
```

**可能的超時情況：**
- 如果 Monday API 回應慢，MaiAgent 會超時
- 大檔案上傳可能超過超時限制
- 不支援背景工作和非同步通知

---

### 2.3 檔案存儲的限制

#### **限制 1：本機存儲不具擴展性**

**問題：** 當前檔案存儲在伺服器本機 `./uploads` 目錄。

**限制：**
- 伺服器重啟會遺失檔案
- 無法橫向擴展到多伺服器
- 無法實現備份和容災
- 本機磁碟空間有限

**生產環境無法接受，需要改用雲存儲（AWS S3, Cloudinary）**

---

#### **限制 2：檔案存取權限**

**問題：** 以 URL 方式存儲的檔案任何人都能存取。

**安全風險：** ⚠️
- 上傳的私密檔案可能被洩露
- 無法實施存取控制
- 無法審計檔案存取

---

### 2.4 認證與授權的限制

#### **限制 1：單一授權使用者**

**問題：** 目前系統用的是**全局 OAuth App**，所有使用者都用同一個 Client ID/Secret。

**限制：**
```
Monday.com 的 OAuth 應用是全局的 (Workspace 級別)
↓
所有使用者授權後都對映到同一個 App
↓
無法區分哪個使用者在什麼時間做了什麼操作 (從 Monday 角度)
```

---

#### **限制 2：無三方登入**

**問題：** 系統要求每個使用者都授權一次。

**限制：**
- 無法像「使用 Google 登入」那樣自動識別使用者
- 每個新使用者都要手動走一遍 OAuth 流程
- 無法透過企業目錄（LDAP/SSO）自動關聯

---

### 2.5 伸縮性與效能限制

#### **限制 1：單伺服器架構**

**當前設計：**
```
localhost:3001 (OAuth)
localhost:3002 (Upload)
localhost:3003 (MCP)
  ↓
All in one Node.js process
```

**伸縮問題：**
- 無法應付高並發
- 單點故障會全服務當機
- Token 存儲在內存，重啟遺失

---

#### **限制 2：內存/磁碟有限**

**當前實現：**
```javascript
// tokens.json 直接讀寫到磁碟
fs.readFileSync('./tokens.json')
fs.writeFileSync('./tokens.json', ...)
```

**問題：**
- 並發寫入會遺失資料
- 大量使用者時 JSON 檔案會很大
- 效能倒退（頻繁 I/O）

**改進方案：** 使用資料庫存儲
