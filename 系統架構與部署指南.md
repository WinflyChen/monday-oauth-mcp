# 系統架構與部署指南

---

## 一、方案 A 完整技術架構

### 整體流程

```
用戶 Telegram
    │
    │ 發送訊息 / 指令
    ▼
telegram-bridge.js（Port 3004）
    │
    ├── /login 指令 → 產生 Monday OAuth 連結 → 導向 server.js
    │
    ├── 一般訊息 → 呼叫 MaiAgent completions API
    │                       │
    │                       ▼
    │               MaiAgent Chatbot
    │                       │
    │                       ▼
    │               MCP Server（Port 3003）← MaiAgent 呼叫
    │                       │
    │                       ▼
    │               Monday.com GraphQL API
    │
    └── OAuth 成功通知 ← server.js（Port 3001）
                │
                ├── 儲存 Monday token（tokens.json）
                ├── 建立 MaiAgent Contact（方案 A 新增）
                └── 通知 telegram-bridge.js 授權完成
```

### 用戶授權流程

```
1. 用戶在 Telegram 發 /login
2. telegram-bridge.js 產生帶有 telegramUserId 的授權連結
3. 用戶點擊連結 → Monday.com 授權頁面
4. 用戶同意授權 → Monday 回傳 code 到 server.js
5. server.js 用 code 換取 access token
6. server.js 將 token 存入 tokens.json（key = mondayUserId）
7. server.js 在 MaiAgent 後台建立 Contact（含 mondayUserId metadata）
8. server.js 通知 telegram-bridge.js 授權完成
9. telegram-bridge.js 更新 session（telegramUserId → mondayUserId 對應）
10. 用戶回到 Telegram，可開始使用
```

### 多用戶支援機制

| 檔案 | 用途 |
|------|------|
| `tokens.json` | 儲存每個用戶的 Monday token（key = mondayUserId） |
| `telegram_sessions.json` | 對應 Telegram ID → Monday ID + 對話 ID |

每個用戶有獨立的 token，Chatbot 呼叫 MCP 工具時帶入各自的 `mondayUserId`，完全隔離。

## 二、本地端服務項目詳細資料

### 服務一：OAuth Server（server.js）

| 項目 | 內容 |
|------|------|
| 檔案 | `server.js` |
| Port | `3001` |
| 對外網址 | Cloudflare tunnel → `https://xxx.trycloudflare.com` |
| 用途 | 處理 Monday.com OAuth 授權流程、儲存 token |

**主要端點：**

| 端點 | 說明 |
|------|------|
| `GET /oauth/authorize` | 產生 Monday 授權連結，帶入 telegramUserId |
| `GET /oauth/callback` | Monday 授權完成後的回調，換 token、建 Contact、通知 Bridge |
| `GET /api/tokens` | 查看目前所有已授權用戶 |
| `GET /health` | 服務健康檢查 |

**OAuth callback 執行的事項：**
1. 用 code 換取 Monday access token + refresh token
2. 取得 Monday 用戶資訊（名稱、Email、User ID）
3. 儲存 token 到 `tokens.json`
4. 在 MaiAgent Contacts 建立此用戶的 Contact（含 `mondayUserId` metadata）
5. 通知 telegram-bridge.js 授權完成

---

### 服務二：Telegram Bridge（telegram-bridge.js）

| 項目 | 內容 |
|------|------|
| 檔案 | `telegram-bridge.js` |
| Port | `3004` |
| 對外網址 | Cloudflare tunnel → `https://xxx.trycloudflare.com`（需另一條） |
| 用途 | 接收 Telegram webhook、轉發至 MaiAgent、管理 session |

**主要端點：**

| 端點 | 說明 |
|------|------|
| `POST /webhook` | Telegram 訊息入口（Bot 的所有訊息都打這裡） |
| `POST /telegram/oauth-success` | 被 server.js 呼叫，更新用戶 session |
| `POST /telegram/set-webhook` | 設定 Telegram Bot webhook URL |
| `GET /telegram/webhook-info` | 查詢目前 webhook 設定狀態 |
| `GET /health` | 服務健康檢查 |

**支援的 Telegram 指令：**

| 指令 | 功能 |
|------|------|
| `/start` | 歡迎訊息 |
| `/login` | 產生 Monday 授權連結 |
| `/status` | 查看目前登入狀態 |
| `/reset` | 清除登入狀態（重新登入） |

**訊息處理邏輯：**
1. 收到訊息 → 從 `telegram_sessions.json` 讀取 session
2. 取得此用戶的 `mondayUserId` 和 `conversationId`
3. 若是新對話，在訊息前加入系統提示（包含 mondayUserId）
4. 呼叫 MaiAgent completions API，帶入 `conversationId`
5. 回傳 MaiAgent 的回覆給用戶
6. 儲存新的 `conversationId` 到 session

---

### 服務三：MCP Server（mcp-server.js）

| 項目 | 內容 |
|------|------|
| 檔案 | `mcp-server.js` |
| Port | `3003` |
| 存取方式 | MaiAgent 透過 HTTP 呼叫（需對外可達） |
| 用途 | 實作 MCP 協議，提供 Monday.com 操作工具給 MaiAgent |

**提供的 MCP 工具：**

| 工具名稱 | 功能 |
|---------|------|
| `monday_get_boards` | 取得用戶所有 Board 清單 |
| `monday_create_board` | 在指定工作區建立新 Board |
| `monday_create_item` | 在 Board 中新增項目 |
| `monday_get_items` | 取得 Board 中的所有項目 |
| `monday_get_auth_url` | 產生 OAuth 授權連結 |
| `monday_upload_file_to_column` | 上傳檔案到 Board 欄位 |

---

### 服務四：Upload Server（upload-server.js）

| 項目 | 內容 |
|------|------|
| 檔案 | `upload-server.js` |
| Port | `3002` |
| 用途 | 處理檔案上傳、提供下載連結 |

---

### 本地端資料檔案

| 檔案 | 內容 | 格式 |
|------|------|------|
| `tokens.json` | 各用戶 Monday token | `{ mondayUserId: { accessToken, refreshToken, expiresAt } }` |
| `telegram_sessions.json` | TelegramID → MondayID 對應 | `{ telegramUserId: { mondayUserId, conversationId, updatedAt } }` |
| `.env` | 所有環境變數（金鑰、Port、URL） | key=value |

---

### 環境變數一覽（.env）

| 變數 | 用途 |
|------|------|
| `MONDAY_CLIENT_ID` | Monday App OAuth Client ID |
| `MONDAY_CLIENT_SECRET` | Monday App OAuth Client Secret |
| `MONDAY_REDIRECT_URI` | OAuth 授權完成後的回調 URL |
| `OAUTH_SERVER_URL` | server.js 的對外網址 |
| `TELEGRAM_BOT_TOKEN` | Telegram Bot API Token |
| `MAIAGENT_API_KEY` | MaiAgent API 金鑰 |
| `MAIAGENT_CHATBOT_ID` | MaiAgent Chatbot UUID |
| `MAIAGENT_API_BASE` | MaiAgent API 基礎網址 |
| `MAIAGENT_MCP_TOOL_ID` | MaiAgent 中 MCP 工具的 UUID |
| `MAIAGENT_INBOX_ID` | MaiAgent Inbox UUID（方案 A 建 Contact 用） |

---

### 本地端啟動方式

```bash
# 啟動所有服務
node server.js           # Port 3001（OAuth）
node telegram-bridge.js  # Port 3004（Telegram）
node mcp-server.js       # Port 3003（MCP）
node upload-server.js    # Port 3002（上傳）

# 啟動 Cloudflare Tunnel（需兩條）
cloudflared tunnel --url http://localhost:3001  # OAuth 用
cloudflared tunnel --url http://localhost:3004  # Telegram webhook 用
```

> ⚠️ **本地端限制**：Cloudflare 臨時 tunnel 每次重啟網址都會變，需要手動更新 Monday OAuth Redirect URL 和 Telegram webhook。

## 三、部署雲端方案比較

### 方案 A：VPS（推薦正式環境）

**適合**：長期穩定運行、多人使用

| 項目 | 說明 |
|------|------|
| 推薦服務商 | DigitalOcean、Vultr、Hetzner |
| 費用 | $4 ～ $6 / 月 |
| Node.js 版本 | 22.x（需手動安裝） |
| 進程管理 | pm2（自動重啟、開機啟動） |
| Reverse Proxy | nginx（統一 443 port，SSL 終止） |
| SSL 憑證 | Let's Encrypt（免費） |
| 固定網址 | ✅ 有（綁定自己的網域） |

**部署步驟概覽：**

```bash
# 1. 在 VPS 安裝 Node.js + pm2 + nginx
# 2. 上傳程式碼（git clone 或 scp）
# 3. 設定 .env（更新為正式網域）
# 4. 用 pm2 啟動所有服務

pm2 start server.js          --name "oauth-server"
pm2 start telegram-bridge.js --name "telegram-bridge"
pm2 start mcp-server.js      --name "mcp-server"
pm2 start upload-server.js   --name "upload-server"
pm2 save                     # 開機自動啟動

# 5. 設定 nginx：將 bot.yourdomain.com/webhook 導向 localhost:3004
# 6. 更新 Monday App 的 Redirect URL → https://bot.yourdomain.com/oauth/callback
# 7. 設定 Telegram webhook → https://bot.yourdomain.com/webhook
```

---

### 方案 B：Railway / Render（PaaS，省管理成本）

**適合**：不想管伺服器、快速上線

| 項目 | Railway | Render |
|------|---------|--------|
| 費用 | $5/月起（有免費額度） | 免費方案有休眠問題 |
| 部署方式 | 連接 GitHub 自動部署 | 連接 GitHub 自動部署 |
| 固定網址 | ✅ 有 | ✅ 有 |
| 多服務 | 每個 js 獨立 Service | 同上 |
| 環境變數 | 後台設定 | 後台設定 |
| 限制 | 需要 4 個 Service | Render 免費版 15 分鐘無流量會休眠 |

**注意**：`tokens.json` 和 `telegram_sessions.json` 在 PaaS 上每次部署會被清空，需改用外部資料庫（如 PostgreSQL 或 Redis）儲存。

---

### 方案 C：Cloudflare Named Tunnel（零成本，本機運行）

**適合**：小規模使用、不想付費、電腦保持開著

| 項目 | 說明 |
|------|------|
| 費用 | 免費 |
| 固定網址 | ✅ 有（需要 Cloudflare 帳號 + 網域） |
| 服務位置 | 仍在本機執行 |
| 可用性 | 電腦關機即中斷 |
| 設定複雜度 | 中（需要 Cloudflare 帳號） |

```bash
# 建立固定 Named Tunnel（只需做一次）
cloudflared tunnel login
cloudflared tunnel create monday-mcp
cloudflared tunnel route dns monday-mcp bot.yourdomain.com

# 設定 config.yml，讓兩個 port 都走同一個網域的不同路徑
# 之後只需執行：
cloudflared tunnel run monday-mcp
```

---

### 三方案綜合比較

| 比較項目 | VPS | PaaS（Railway） | Cloudflare Named Tunnel |
|---------|:---:|:---:|:---:|
| 月費 | $4 ～ $6 | $5 起 | 免費 |
| 固定網址 | ✅ | ✅ | ✅（需網域） |
| 24h 可用性 | ✅ | ✅ | ❌（電腦需開著） |
| 資料持久性 | ✅ | ⚠️（需改資料庫） | ✅（本機檔案） |
| 管理難度 | 中 | 低 | 低 |
| 擴充彈性 | 高 | 中 | 低 |
| 推薦情境 | 正式環境 | 快速上線 | 內部小規模測試 |

---

### 正式上線前的必要調整

無論選擇哪個雲端方案，都需要：

1. **更新 Monday App Redirect URL**  
   Monday.com 開發者後台 → OAuth → Redirect URLs → 改為正式網域

2. **重設 Telegram Webhook**  
   ```bash
   curl -X POST "http://localhost:3004/telegram/set-webhook" \
     -H "Content-Type: application/json" \
     -d '{"url": "https://bot.yourdomain.com"}'
   ```

3. **更新 .env 所有 URL 相關變數**  
   - `MONDAY_REDIRECT_URI`
   - `OAUTH_SERVER_URL`
   - `SERVER_URL`

4. **考慮 tokens.json 的持久化**（PaaS 必須）  
   改用 PostgreSQL / Redis / Supabase 取代本機 JSON 檔案

## 四、Monday.com App 多機構用戶授權說明

### 背景

本系統使用的 Monday App 是**私有（Private）App**，預設只有建立該 App 的機構成員可以授權使用。若要讓其他機構的用戶使用，需要額外處理。

### 方案一：邀請用戶加入你的機構

**適合**：內部測試、少數指定人員

| 步驟 | 操作 |
|------|------|
| 1 | Monday.com → 右上角邀請按鈕 → **Invite to work management** |
| 2 | 輸入對方 Email，選擇角色後點 **Invite** |
| 3 | 對方接受邀請後，走 `/login` 流程即可授權 |

**可用角色與限制：**

| 角色 | 讀取 | 寫入 | 適合情境 |
|------|:----:|:----:|---------|
| Member | ✅ | ✅ | 內部員工 |
| Viewer | ✅ | ❌ | 只需要查詢資料的外部人員 |
| Guest | ✅ | ✅（受限範圍） | 需要 Pro 以上方案才有此選項 |

> ⚠️ **注意**：用戶加入後使用的是**你的機構**資料，不是他自己原本的 Monday 工作區。

---

### 方案二：Promote to live + Share app（推薦多機構場景）

**適合**：用戶有自己獨立的 Monday.com 機構、各自操作自己的資料

**步驟：**

1. Monday 開發者後台 → **Distribute → Share app**
2. 勾選「I accept monday.com Developer Terms」→ 點 **Publish app**
3. 點左上角 **Promote to live**（App 狀態從 Draft → Live，仍是 Private，不會上架 Marketplace）
4. 回到 Share app 頁面，複製**安裝連結**
5. 將安裝連結傳給對方機構的**管理員**
6. 管理員點連結 → 用管理員帳號登入 → 點 Install → App 安裝到該機構
7. 該機構所有成員之後走 `/login` 流程即可授權，使用自己的資料

> ✅ 各機構資料完全隔離，每個用戶操作的是自己機構的 boards 和 items。

---

### 兩方案比較

| 比較項目 | 方案一（邀請加入） | 方案二（Share app） |
|---------|:---:|:---:|
| 操作複雜度 | 低 | 中 |
| 用戶使用自己的資料 | ❌（使用你的機構資料） | ✅ |
| 需要 Promote to live | ❌ | ✅ |
| 適合人數 | 少數測試人員 | 多機構、正式推廣 |
| 用戶寫入權限 | 依角色而定 | ✅ 完整權限 |
